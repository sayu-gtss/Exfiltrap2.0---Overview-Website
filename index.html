The Problem

Why DNS Security Matters

DNS is the backbone of the internet — and one of the most exploited attack vectors. Existing solutions lack real-time detection, contextual intelligence, and scalable handling of large DNS log volumes.

DNS Tunneling

Attackers encode sensitive data into DNS queries to silently exfiltrate information, bypassing firewalls and perimeter defenses without raising alarms.

DGA-Based Malware

Malware uses Domain Generation Algorithms to generate thousands of pseudo-random domains daily, making blacklisting-based defenses ineffective.

DNS Infrastructure Abuse

Open resolvers are exploited for amplification attacks, creating massive traffic floods that overwhelm victim networks and disrupt services.

Our Solution

ExfilTrap — What We Built

A unified, six-layer detection pipeline that combines ML inference, behavioral analysis, and contextual threat intelligence to detect and explain DNS-based threats in near real-time.

Unified Sniffer Agent

Lightweight C++ endpoint agent leveraging Npcap for passive DNS capture across UDP/TCP port 53, TCP port 443 (DoH), and TCP port 853 (DoT). Privacy-preserving — analyzes metadata without decryption.

ML Detection Engine

Three parallel subsystems handle DNS tunneling, DGA-based C2 activity, and infrastructure abuse. Uses Logistic Regression, Random Forest, and Dempster–Shafer Theory for probabilistic fusion.

Threat Intelligence Module

Continuous enrichment pipeline ingesting VirusTotal, dark source feeds, and OSINT data. Applies risk scoring and sector classification to every detection event.

SIEM / SOC Dashboard

Real-time visibility through Kibana integration. Alert queues, severity distributions, and analyst-friendly investigation interface with explainability support for every alert.

Literature Survey

Overview of the Domain Name System (DNS)

The Domain Name System (DNS) resolves domain names into IP addresses through a hierarchical query process involving multiple distributed servers. The resolution process involves DNS query initiation, root server queries, TLD server queries, authoritative server queries, and final response delivery to the client.

DNS traffic exhibits several distinct characteristics: it is high-volume and continuous, lightweight and structured, and highly repetitive. While caching at recursive resolvers reduces repeated lookups, client devices continue to generate recurring queries due to session expiry, load balancing, and application behavior.

Security Assumptions and Historical Design Limitations

DNS was originally designed as a high-performance, available system without security as a core objective. It lacks built-in mechanisms to verify query authenticity or restrict protocol misuse. While DNSSEC provides record integrity validation, it cannot prevent DNS from being exploited as a covert communication channel.

The openness of domain registration and global accessibility of recursive resolvers make DNS an attractive attack vector. Malicious DNS activity closely mimics legitimate traffic patterns, and the protocol's request/response brevity allows data or commands to be concealed within seemingly normal queries.

DNS as an Attack Vector

DNS traffic is often implicitly trusted within enterprise networks. Attackers exploit this trust to embed malicious communication within seemingly legitimate DNS queries — bypassing firewall restrictions, evading intrusion detection systems, and maintaining persistent communication with compromised hosts.

Classification of Existing Detection Approaches

  • Lexical and structural methods: Analyze domain name patterns for anomalies
  • Statistical and traffic volume methods: Analyze DNS traffic metrics over time
  • Behavioral and temporal methods: Analyze timing and interaction patterns
  • Machine learning-based methods: Use trained models to classify DNS activity
  • Threat intelligence-driven methods: Use external intelligence to identify known malicious indicators

Research Gap

The vast majority of existing literature tests models using synthetic datasets, pre-collected data in controlled laboratory settings, or simulated traffic — insufficiently representative of live DNS environments.

While evaluation settings that are reproducible have advantages, they are insufficient for real-world deployments due to additional factors including incremental feature extraction, adaptive baselining, and continuous inference in a streaming context.

Furthermore, the increasing adoption of DNS over HTTPS (DoH) and DNS over TLS (DoT) has significantly reduced visibility into query payloads, complicating detection of data exfiltration by obscuring query contents and forcing analysts to rely on metadata and behavioral indicators.

Current CTI platforms also suffer from "Alert Fatigue" — large amounts of generic indicators that lack organizational context — making it difficult for SOC analysts to effectively triage threats.

Research Problem

Existing DNS security solutions lack near real-time detection capabilities, contextual threat intelligence, and efficient handling of the large DNS log volumes required for ML-based analysis in enterprise environments.

Specifically, organizations face three compounding challenges:

  • Large volumes of DNS traffic must be collected and stored before being processed by ML models, creating significant storage requirements and data management challenges.
  • Encrypted DNS (DoH/DoT) adoption limits payload inspection, forcing detection to rely on metadata alone.
  • Existing threat intelligence platforms deliver generic alerts without organizational context, contributing to analyst alert fatigue in SOC environments.

Research Objectives

Main Objective: Develop an ML-based near real-time DNS anomaly detection system with centralized monitoring and contextual threat intelligence for enterprise environments.

  • Design and implement a unified DNS packet capture agent that handles plaintext and encrypted DNS traffic (DoH/DoT) without compromising user privacy.
  • Develop parallel ML-based detection subsystems for DNS tunneling, DGA-based malware, C2 communication, and DNS infrastructure abuse.
  • Build a contextual threat intelligence and resilience module that enriches detection events with external IoCs, risk scores, and sector classifications.
  • Integrate with enterprise SIEM environments (Kibana) to deliver real-time visibility and analyst-friendly investigation workflows.
  • Provide explainability support for all detection decisions to enable effective human oversight.

Methodology

System Architecture

The system follows a producer–consumer architecture that decouples packet capture from ML inference to prevent processing bottlenecks. The high-level architecture comprises 6 principal layers arranged in a sequential processing pipeline:

  1. Unified Sniffer Agent — Lightweight C++ endpoint agent for passive DNS capture via Npcap
  2. Detection Engine — Three parallel ML subsystems targeting distinct threat categories
  3. Threat Intelligence & Resilience Module — Continuous enrichment pipeline with external feeds
  4. Centralized Relational Database — MySQL integration backbone for all data
  5. Query Interface & Rule Management — Configurable rule engine with custom detection logic
  6. Operational Dashboard & SIEM Integration — Real-time Kibana monitoring for SOC analysts

Data Acquisition

Data sources include public datasets (CICBellEXFDNS, Kaggle DNS exfiltration, CIC DDoS, IOT, DOHbwr for lexical classification, and VCDB benign Alexa top 1M), live traffic captured via the deployed sniffer agent, and lab-generated malicious traffic using Iodine, dnscat2 frameworks, and custom exfiltration scripts.

Detection Engine Design

The detection engine employs specialized feature engineering per subsystem. Dempster–Shafer Theory (DST) is used for probabilistic fusion of lexical and behavioral model outputs to handle uncertainty and conflicting predictions. TF-IDF vectorization over character-level n-grams captures lexical patterns of DGA domains.
